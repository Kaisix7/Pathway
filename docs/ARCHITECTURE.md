# Архитектура PATHWAY

## Слои проекта

PATHWAY разделён на три понятных слоя:

- Flutter frontend: экраны, навигация, формы, валидация, загрузка и локальное состояние приложения.
- Django backend: HTTP API, модели базы данных, регистрация, логин, заказы, оплата и аналитика.
- База данных: постоянное хранение пользователей, заказов и событий аналитики.

## За что отвечает Flutter

Flutter отвечает за пользовательский интерфейс:

- onboarding и выбор роли
- экраны сервисов
- поиск на карте через OpenStreetMap/Nominatim
- форма заказа Airport
- экран оплаты
- страницы аккаунта: About, Help, Terms of Service, Privacy Policy

Flutter общается с Django через файл `lib/api_service.dart`.

## За что отвечает Django

Django отвечает за данные и бизнес-логику:

- регистрация пользователя
- логин по email
- связь пользователя с заказами
- создание и получение заказов
- изменение статуса заказа с `pending` на `done` после оплаты
- простая аналитика через события и endpoint `/api/metrics/`

Маршруты backend находятся в `backend/core/urls.py`.

## База данных

В Docker используется PostgreSQL.

Если запускать backend без Docker, проект может использовать SQLite. Это сделано как запасной вариант для локальной разработки.

Основные таблицы:

- `AppUser`: пользователь приложения, email, имя, тариф подписки.
- `AirportOrder`: заказ или сервисная заявка, связанная с пользователем.
- `AppEvent`: событие аналитики, например регистрация, логин, создание заказа или оплата.

## Жизненный цикл заказа

1. Пользователь регистрируется или входит в приложение.
2. Flutter создаёт заказ через `POST /api/orders/`.
3. Backend сохраняет заказ со статусом `pending`.
4. Пользователь открывает Payments и оплачивает заказ.
5. Flutter вызывает `POST /api/orders/<id>/pay/`.
6. Backend меняет статус заказа на `done`.

## API

Основные endpoints:

- `POST /api/register/`
- `POST /api/login/`
- `GET /api/orders/`
- `POST /api/orders/`
- `POST /api/orders/<id>/pay/`
- `POST /api/events/`
- `GET /api/metrics/`
- `GET /api/openapi.json`
